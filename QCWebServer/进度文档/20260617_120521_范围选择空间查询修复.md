# 范围选择空间查询修复

## 本次目标
- 排查并修复范围选择、自定义范围选择依旧无法返回数据的问题。
- 使用多 agent 协同信息：复用已完成子代理 Singer 的只读结论；本轮尝试新建后端调查子代理时因 agent thread limit reached 失败，已由主代理继续完成实现。
- 完成后生成进度文档，并提交推送到 main 分支。

## 做了哪些
- 将范围选择从依赖 OpenLayers MVT 渲染瓦片缓存，改为优先调用服务端 PostGIS 空间查询。
- 新增 POST /api/layers/:id/features/select，接收 EPSG:4326 GeoJSON geometry 和 limit，使用 ST_Intersects 查询相交要素主键。
- 前端将框选矩形和自定义绘制多边形的 EPSG:3857 几何序列化为 EPSG:4326 GeoJSON 后提交接口。
- 查询到 ids 后复用已有属性表 ids 过滤能力，打开范围内要素属性表。
- 保留已渲染瓦片采样和旧 getFeaturesInExtent 作为接口不可用时的兜底路径。
- 增加前端单测覆盖选择几何坐标转换和采样兜底工具。
- 已执行验证：`npm run typecheck`、`npm --workspace apps/web run test`、`npm --workspace apps/api run test`，均通过。

## 没做哪些
- 未启动真实 PostGIS 数据库做端到端人工框选验证；当前验证覆盖类型检查、单元测试和接口链路编译正确性。
- 未处理本次任务无关的既有未提交改动，包括根目录 README.md 删除、设计稿 下文件删除、旧进度文档修改。

## 哪里有 bug
- 根因 bug：范围选择依赖 VectorTileLayer#getFeaturesInExtent() 从 OpenLayers 当前 renderer tile cache 取已加载瓦片，框选范围跨未加载瓦片或当前 zoom 缓存不完整时会返回空，因此范围选择和自定义范围选择不稳定。
- 已修复：新增数据库空间查询后，范围选择不再依赖 MVT 缓存枚举。
- 残留风险：后端空间查询假设前端传入 geometry 为 EPSG:4326 GeoJSON；若未来支持非 EPSG:3857 地图视图，需要同步调整序列化投影参数。

## 建议后续做什么
- 增加带测试数据库的 API 集成测试，实际插入点线面要素验证 `features/select` 返回 ids。
- 在 UI 上补充“命中超过 500 条，仅显示前 500 条”的提示，目前接口返回 total 但前端只消费 ids。
- 如果需要跨图层范围选择，可扩展为按可见图层批量查询并合并结果；当前优先查询可见活动 PostGIS 图层。
