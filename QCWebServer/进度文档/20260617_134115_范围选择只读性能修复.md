# 范围选择、只读与大表性能修复

## 本次目标
- 排查并修复范围选择、自定义范围选择仍无法返回数据的问题。
- 排查新增点线表显示只读的原因，并让可操作表进入可查询/可编辑状态。
- 优化 90w+ 点线表首屏展示慢的问题。
- 使用多 agent 协同；本轮成功使用 Raman 做只读调查，结论与主流程验证一致。
- 完成后生成进度文档，并提交推送到 main 分支。

## 做了哪些
- 后端扫描 PostGIS 图层时，不再只依赖 geometry_columns；当 SRID 或几何类型缺失时，会从真实几何抽样读取 ST_SRID 与 GeometryType。
- 对没有单字段主键但具备 SELECT/INSERT/UPDATE/DELETE 权限且 SRID 有效的表，允许用 PostgreSQL ctid 作为行标识兜底，支持查询、范围选择、属性表展示和编辑链路。
- 修复范围选择 SQL 中 selection.geom 与业务表 geom 字段同名导致的歧义问题。
- 修复 ST_Transform 使用参数占位符导致 PostGIS 将 SRID 误判为 proj string 的问题。
- 优化 MVT 瓦片 SQL：将瓦片 bounds 反投影到图层 SRID 后用 geom && bounds + ST_Intersects 命中空间索引，再对候选要素做 ST_Transform 输出 MVT。
- 优化扫描 extent：使用 ST_EstimatedExtent，并将估算范围转为 EPSG:4326，避免扫描 90w+ 全表计算 ST_Extent。
- 优化属性表首屏总数：无筛选时使用 pg_class.reltuples 估算行数，避免默认打开属性表时 count(*) 全表扫描。
- 前端统一将 sourceType 缺省的后端图层视为 PostGIS，避免 queryable 图层被严格 sourceType 判断误拦截。
- 新增/更新单元测试覆盖 ctid 兜底图层、sourceType 缺省判断、SQL helper 行为。
- 使用真实本地 API 和数据库验证：扫描后 sys_016_tancedbtjinfo_sde / sys_016_tancexbtjinfo_sde 为 SRID 3857、POINT/LINESTRING、queryable=true、editable=true；属性表首屏返回 5 条；features/select 返回 ctid ids。
- 使用 Chrome 插件打开 http://127.0.0.1:5173 验证工作台可加载，展开数据源后两张目标表在 UI 中显示“可编辑”。

## 没做哪些
- 没有给数据库表真正新增主键或唯一约束；当前采用 ctid 作为无主键表的行标识兜底。
- 没有把运行时扫描刷新后的 apps/api/data/layers.json 纳入提交；用户在现有环境需要重新扫描数据源以刷新图层元数据。
- 没有做大规模可视化帧率基准，只做了接口级和浏览器 UI 健康验证。
- 没有处理本任务无关的既有未提交改动：README.md 删除、设计稿文件删除、旧进度文档修改。

## 哪里有 bug
- 根因 1：新增点线表的 geometry_columns 元数据没有记录 SRID，原扫描逻辑得到 srid:null，且只认正式单字段主键，导致 queryable=false/editable=false。
- 根因 2：范围选择前端会跳过 queryable=false 图层，后端也要求有效 SRID 和主键；所以新增点线表无法返回范围选择结果。
- 根因 3：上一版空间选择 SQL 使用 selection.geom，和业务表常见 geom 字段同名时会出现 column reference ambiguous。
- 根因 4：MVT 查询在 WHERE 中对表字段做 ST_Transform(t.geom, 3857)，会削弱/绕过 geom 空间索引，90w+ 表首屏瓦片容易慢。
- 根因 5：扫描 ST_Extent(ST_Transform(...)) 和属性表 count(*) 会对大表造成明显首屏/扫描压力。

## 建议后续做什么
- 数据库层最好为 sys_016_tancedbtjinfo_sde、sys_016_tancexbtjinfo_sde 增加稳定主键或唯一约束；ctid 适合作为兜底，不适合作为长期业务 ID。
- 在 UI 上提示“无主键表使用临时行标识”，避免用户误解 ctid 的长期稳定性。
- 为大表建立或确认 geom 的 GiST/SP-GiST 索引，并定期 ANALYZE，保证 ST_EstimatedExtent 和 reltuples 足够准确。
- 搜索功能后续可改为字段级搜索或 pg_trgm/全文索引，避免 to_jsonb(t)::text ilike 在大表上全表扫。
- 对 features/select 的 total 可进一步改为“hasMore/returned”，避免大范围选择为了精确总数产生额外成本。
