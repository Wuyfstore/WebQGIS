# 类 QGIS WebGIS V1 项目计划

目标：构建一个面向 PostGIS 矢量数据源的 Web 端图层浏览、编辑与自动矢量瓦片发布平台。

版本：V1 计划草案 日期：2026-06-11 适用阶段：MVP / 技术预研 / 第一版产品定义

> **执行结论**
> 该 V1 范围可行，建议定位为“Web PostGIS 编辑与发布平台”，而不是完整 Web 版 QGIS。第一版应严格限定数据源、编辑对象和吸附范围：只支持 PostGIS geometry 表，矢量瓦片只用于展示，编辑时回源读取原始几何并保存回库。

## 1. 项目定位

本项目第一版不是复刻完整 QGIS 桌面能力，而是建立一个可持续扩展的 Web GIS 内核：连接 PostgreSQL/PostGIS 数据源，自动发现空间表，生成矢量瓦片服务用于地图展示，并提供点、线、面数据的几何编辑、属性编辑和基础吸附能力。

第一版的成功标准不是覆盖所有 GIS 场景，而是把“连接数据源、发布图层、浏览地图、编辑要素、写回数据库”这条主流程做稳定。后续再扩展高级样式、拓扑规则、全库吸附、空间分析、打印制图和多用户协作。

## 2. V1 范围边界

### 2.1 必做范围

- 连接 PostgreSQL/PostGIS 数据源，保存连接配置并进行权限隔离。
- 扫描 schema、table、geometry column、SRID、主键、字段、空间索引和几何类型。
- 自动注册可展示图层，并为每个空间图层生成矢量瓦片访问地址。
- 地图端加载矢量瓦片，支持图层开关、排序、透明度和基础样式。
- 支持 Point、LineString、Polygon 及 Multi* 类型的要素选择、绘制、节点编辑和删除。
- 支持属性表查看、单要素属性面板编辑和字段类型映射。
- 支持当前视野或当前编辑图层内的基础吸附。
- 保存编辑结果回 PostGIS，并在保存后刷新对应图层展示。

### 2.2 暂不纳入 V1

- 完整 QGIS 插件体系、模型构建器和复杂制图打印。
- 任意文件格式导入导出，例如 Shapefile、GeoPackage、DXF、GeoTIFF 的全量兼容。
- 完整 QGIS 样式系统，例如复杂规则渲染、表达式符号、标注避让和图例联动。
- 跨全库、跨图层、跨比例尺的高性能服务端吸附。
- 拓扑规则引擎，例如面不能重叠、线必须相接、节点必须共点等。
- 复杂空间分析，例如缓冲区、叠加、裁剪、栅格计算和网络分析。
- 多用户协同编辑、版本管理、冲突合并和审计工作流。

## 3. 推荐总体架构

架构建议采用“前端 GIS 编辑器 + 后端数据源管理 API + PostGIS + 矢量瓦片服务”的方式。展示和编辑需要分离：矢量瓦片只负责快速展示；编辑时通过主键回源读取原始几何和属性，保存时直接写回 PostGIS。

| 层级 | 推荐选型 | 职责 |
| --- | --- | --- |
| 前端地图 | OpenLayers | 地图渲染、选择、绘制、Modify 节点编辑、Snap 吸附、图层交互。 |
| 前端框架 | Vue3 | 承载 QGIS 风格 UI、状态管理、属性面板、图层树和工具栏。 |
| 后端 API | NestJS 连接管理、图层注册、要素 CRUD、权限、样式配置、编辑校验。 |
| 空间数据库 | PostgreSQL + PostGIS | 存储矢量数据，提供空间索引、空间查询、几何校验和坐标转换。 |
| 瓦片服务 | Martin / pg_tileserv 思路 / 自研 ST_AsMVT | 自动把 PostGIS 空间表发布为 MVT 矢量瓦片服务。 |
| 缓存与任务 | Redis 可选 | 缓存瓦片、记录刷新状态、承载后续异步分析任务。 |

## 4. 核心设计原则

- 展示链路和编辑链路分离：MVT 用于展示，原始几何用于编辑和保存。
- 只把满足编辑条件的空间表标记为可编辑：必须有主键、geometry 字段、SRID 和可写权限。
- V1 优先稳定支持 geometry 类型，不把 geography、无主键视图、复杂表达式视图作为编辑对象。
- 吸附先做当前视野内的客户端吸附，服务端全库吸附作为 V1.5 或 V2 能力。
- 图层注册表是核心资产，所有自动发布、编辑权限、样式和瓦片 URL 都围绕它管理。

## 5. 关键模块拆解

| 模块 | V1 功能 | 关键难点 |
| --- | --- | --- |
| 数据源连接 | 连接 PostgreSQL/PostGIS，保存连接配置，测试连通性。 | 凭据安全、连接池隔离、不同 schema 权限处理。 |
| 空间表扫描 | 识别 geometry 字段、SRID、主键、字段类型、范围和索引。 | 无主键表、多个 geometry 字段、混合几何类型、视图可编辑性判断。 |
| 图层注册 | 生成图层元数据、可编辑状态、默认样式和瓦片 URL。 | 元数据变化后的同步策略和用户自定义配置覆盖。 |
| 矢量瓦片 | 按 z/x/y 输出 MVT，支持视野裁剪、简化和属性筛选。 | 大表性能、空间索引、瓦片缓存、属性字段泄露控制。 |
| 地图展示 | 加载底图和业务图层，支持开关、排序、透明度和点击查询。 | 瓦片样式一致性、图层刷新和大规模要素渲染。 |
| 几何编辑 | 新增、修改、删除点线面，支持节点级编辑。 | Multi 几何、洞、多边形合法性、撤销重做、保存失败回滚。 |
| 属性编辑 | 属性面板和属性表编辑，按字段类型渲染输入控件。 | 字段约束、枚举、空值、日期、数值精度和只读字段。 |
| 吸附能力 | 对当前视野内已加载要素进行顶点和边吸附。 | 吸附性能、图层过滤、比例尺阈值、编辑体验。 |
| 保存校验 | 写回前检查几何合法性、SRID、字段类型和权限。 | 事务处理、并发覆盖、错误提示、坏几何修复策略。 |

## 6. 数据表可编辑条件

建议后端扫描图层后，为每个图层生成状态：可展示、可查询、可编辑。V1 中可编辑图层至少满足以下条件：

- 表或可编辑视图存在单字段主键，且主键值稳定。
- 存在一个明确的 geometry 字段，几何类型属于 Point、LineString、Polygon 或 Multi*。
- geometry 字段有 SRID，且能转换到地图展示坐标系。
- 数据库用户具备 SELECT、INSERT、UPDATE、DELETE 权限中的对应权限。
- 存在空间索引，或系统可以提示用户创建空间索引后再启用高性能展示。
- 字段类型能映射到前端表单控件；无法安全编辑的字段标记为只读。

## 7. 编辑工作流

1. 用户在图层树中选择可编辑图层，并进入编辑模式。
1. 地图展示仍然使用矢量瓦片；用户点击要素时，前端拿到图层 ID 和要素主键。
1. 后端通过图层注册信息回源查询原始几何和属性，返回 GeoJSON 或 WKB 转换结果。
1. 前端把原始几何放入编辑图层，用户进行节点修改、绘制、删除或属性修改。
1. 保存时后端开启事务，校验几何合法性、SRID、字段类型、权限和并发版本。
1. 写回成功后刷新编辑图层和相关瓦片；失败时保留前端草稿并展示可理解错误。

## 8. 里程碑计划

| 阶段 | 周期建议 | 交付物 | 验收标准 |
| --- | --- | --- | --- |
| M0 技术预研 | 1-2 周 | OpenLayers 编辑原型、PostGIS 扫描脚本、MVT 原型。 | 能展示一张 PostGIS 表，点击要素并回源读取原始数据。 |
| M1 数据源与图层注册 | 2-3 周 | 连接管理、schema 扫描、图层注册表、可编辑性判断。 | 能连接多个 PG 数据源并自动列出空间图层。 |
| M2 矢量瓦片发布 | 2-3 周 | MVT 服务、样式配置、图层展示、刷新机制。 | 大表按瓦片加载不卡顿，图层开关和基础样式可用。 |
| M3 要素查询与属性编辑 | 2 周 | 点击查询、属性面板、属性表、字段类型映射。 | 能查看和修改属性，字段校验和错误提示清晰。 |
| M4 几何编辑与吸附 | 3-5 周 | 点线面绘制、节点编辑、删除、客户端吸附。 | 点线面编辑稳定，吸附对当前视野图层可用。 |

## 9. 主要风险与应对

| 风险 | 影响 | V1 应对策略 |
| --- | --- | --- |
| “任意 PG 数据源”过宽 | 导致大量不可编辑边界情况。 | 限定为 PostGIS geometry 表；无主键、无 SRID、无权限时只读展示。 |
| 把 MVT 当作编辑数据源 | 几何会被裁切、简化、量化，写回不可靠。 | MVT 仅用于展示；编辑必须回源读取原始几何。 |
| 大数据量性能不稳定 | 地图卡顿或瓦片响应慢。 | 强制空间索引检查，使用 ST_AsMVTGeom、视野裁剪、属性白名单和缓存。 |
| 吸附范围扩大过快 | 性能和体验不可控。 | V1 只做当前视野或当前图层客户端吸附；服务端吸附后置。 |
| 几何合法性复杂 | 保存后产生坏数据或拓扑错误。 | 写回前使用 ST_IsValid 等校验，必要时阻止保存并提示问题位置。 |
| 并发编辑冲突 | 多人同时修改同一要素会互相覆盖。 | V1 可用 updated_at/version 字段做乐观锁；无版本字段时提示覆盖风险。 |

## 10. V1 验收清单

- 可以新增一个 PostgreSQL/PostGIS 数据源并测试连接。
- 可以自动发现空间表，显示几何类型、SRID、主键和是否可编辑。
- 可以自动发布并加载矢量瓦片图层。
- 可以在地图上选择要素并查看原始属性。
- 可以编辑点、线、面几何并保存回库。
- 可以修改属性字段并进行基础类型校验。
- 可以对当前视野内的要素顶点或边进行吸附。
- 无主键、无权限、无 SRID 或非 geometry 图层不会被误标为可编辑。
- 保存失败时不会破坏数据库原始数据，并能给出明确错误提示。
- 刷新后可以看到保存后的最新图层效果。

## 11. 建议的下一步

建议先用一套示例 PostGIS 数据库做两周技术原型：一张点表、一张线表、一张面表，每张表包含主键、属性字段、空间索引和 SRID。原型阶段优先验证 OpenLayers 编辑体验、MVT 发布性能、回源编辑链路和保存事务。

如果原型通过，再进入产品化阶段，补齐数据源管理、图层注册表、权限体系、错误处理、UI 框架和测试用例。这样可以避免一开始陷入完整 QGIS 能力复刻，先把最核心的 Web GIS 数据编辑闭环做稳。
