# 大数据矢量编辑性能优化实施方案

## 背景

WebQGIS 的目标不是只浏览大数据表，而是要像 QGIS 一样能选择、编辑、保存 PostGIS 矢量数据。90w+ 要素如果全部作为前端可编辑 Feature 加载，交互会很快失控；如果只做离线瓦片，又会牺牲编辑后的即时反馈。

因此推荐采用：

> 浏览层用瓦片保证速度，编辑层只加载少量精确 GeoJSON，保存后用覆盖层即时反馈，再对受影响瓦片做增量刷新。

这份文档用于交给 Codex agent 分阶段实现。

## 当前代码基础

- 前端地图核心：`apps/web/src/composables/useOpenLayersEditor.ts`
- 前端工作区状态：`apps/web/src/composables/useWebGisWorkspace.ts`
- 前端工作台组件：`apps/web/src/components/workbench/WebGisWorkbench.vue`
- 服务端图层接口：`apps/api/src/layers/layers.controller.ts`
- 服务端图层服务：`apps/api/src/layers/layers.service.ts`
- PostGIS 仓储：`apps/api/src/postgis/postgis.repository.ts`
- 类型定义：`apps/web/src/types/gis.ts`、`apps/api/src/types.ts`

当前关键路径：

- MVT 浏览层：`getVectorTile()` 使用 `ST_AsMVTGeom` / `ST_AsMVT` 实时生成瓦片。
- 单要素编辑：点击瓦片要素后 `readFeature()` 回源读取 GeoJSON，再进入 `editSource`。
- 范围选择高亮：`/features/select` 返回 ids/features，前端写入 `selectedFeatureSource`。
- 保存编辑：`saveFeature()` 调用 `POST/PUT /features`，保存成功后调用 `editor.refreshLayer(activeLayerId)` 刷新图层。

## 总体架构

### 1. 浏览层

浏览层只负责快速显示大量数据，不承担编辑状态。

- 数据源：MVT、缓存 MVT、PMTiles、MBTiles 或栅格瓦片。
- 前端承载：`VectorTileLayer` / `VectorTileSource`。
- 目标：平移缩放快，可按 zoom 看到合理细节。

### 2. 选择高亮层

选择高亮层负责显示当前选择结果。

- 数据源：少量 GeoJSON Feature。
- 前端承载：`selectedFeatureSource` / `selectedFeatureLayer`。
- 目标：点击、范围、自定义范围选择后只有一套清晰高亮。

### 3. 编辑覆盖层

编辑覆盖层只负责正在编辑的少量要素。

- 数据源：原始表回源读取的精确 GeoJSON。
- 前端承载：`editSource` / `editLayer`。
- 目标：节点编辑、绘制、保存时精确响应。

### 4. 增量刷新层

编辑保存后，不等待整张表或全图层刷新。

- 前端先更新覆盖层，马上看到结果。
- 后端记录或计算受影响瓦片。
- 前端只刷新当前视野内受影响瓦片，或刷新当前图层但带版本 key。

## 阶段 0：状态边界梳理

### 目标

先把“浏览、选择、编辑”三个状态边界理清楚，为后续性能优化打底。

### 需要实现

1. 明确三类 source 的职责：
   - `VectorTileSource`：浏览大数据。
   - `selectedFeatureSource`：选择结果高亮。
   - `editSource`：当前编辑草稿。
2. 增加统一清理方法：
   - `clearSelectionVisuals(options)`
   - `clearEditDraft(options)`
   - 或组合成 `setSelectionState(kind, payload)`。
3. 点击选择和范围选择切换时，确保旧高亮不会叠加：
   - 范围选择成功后清 `selectedFeatureId`。
   - 范围选择成功后清非编辑态 `editSource`。
   - 点击选择后清范围选择高亮。
4. 如果存在未保存草稿，范围选择或切换图层前应提示保存/放弃。

### 建议改动文件

- `apps/web/src/composables/useOpenLayersEditor.ts`
- `apps/web/src/composables/useWebGisWorkspace.ts`
- `apps/web/src/components/workbench/WebGisWorkbench.vue`
- `apps/web/src/composables/useOpenLayersEditor.test.ts`
- `apps/web/src/components/workbench/WebGisWorkbench.test.ts`

### 验收标准

- 点击选择一个要素后出现单一高亮。
- 再做范围选择，点击高亮消失，只保留范围结果高亮。
- 范围选择后再点击选择，范围高亮消失，只保留点击高亮。
- 未保存编辑草稿不会被静默清掉。

## 阶段 1：编辑后即时反馈

### 目标

保存编辑后，不等瓦片刷新，用户立即看到新几何或新属性状态。

### 需要实现

1. 保存接口继续返回保存后的完整 GeoJSON Feature。
   - 当前 `createFeature()` 和 `updateFeature()` 已经返回 `readFeature()` 结果，可以沿用。
2. 前端 `saveFeature()` 保存成功后：
   - 更新 `selectedFeatureId`。
   - 更新 `selectedProperties`。
   - 更新 `draftGeometry`。
   - 让 `editSource` 中的当前 Feature 替换为服务端返回的 Feature。
3. 新增编辑覆盖状态：
   - `editedFeatureOverlayIds: Map<layerId, Set<featureId>>`
   - 或更简单：`coveredFeatureIds`。
4. MVT 样式函数遇到被覆盖 id 时：
   - 可以暂时透明化旧瓦片要素。
   - 或降低透明度，避免旧几何和新几何重叠造成误解。

### 建议改动文件

- `apps/web/src/composables/useWebGisWorkspace.ts`
- `apps/web/src/composables/useOpenLayersEditor.ts`
- `apps/web/src/components/workbench/WebGisWorkbench.vue`
- `apps/web/src/composables/useOpenLayersEditor.test.ts`

### 关键设计

保存成功后的视觉优先级：

1. `editLayer` 显示最新服务端返回几何。
2. `selectedFeatureLayer` 显示选择结果。
3. MVT 浏览层显示缓存瓦片，但被覆盖 id 可隐藏或弱化。

### 验收标准

- 编辑一个要素并保存后，几何立即显示为新形状。
- 保存成功后不需要等待 MVT 请求完成才能看到修改结果。
- 若旧瓦片未刷新，旧几何不会明显覆盖新几何。

## 阶段 2：图层版本与瓦片缓存

### 目标

让 MVT 能缓存，同时编辑后能快速失效，避免每次视图变化都实时打 PostGIS。

### 需要实现

1. 服务端为每个 PostGIS 图层维护 `tileVersion`。
   - 最小实现可以存在内存 Map：`Map<layerId, number>`。
   - 更可靠实现存在图层 registry 或持久化元数据。
2. 图层返回给前端时带上 `tileVersion`。
3. 前端构建 `tileUrl` 时附加版本：
   - `/api/layers/:id/tile/{z}/{x}/{y}.mvt?v=:tileVersion`
4. 编辑成功后：
   - 后端递增当前图层版本。
   - 保存接口返回 `{ feature, tileVersion }` 或响应 header。
   - 前端更新该图层 tileUrl 或刷新 source。
5. MVT 响应增加缓存头：
   - `Cache-Control`
   - `ETag`
   - 可选：`Last-Modified`
6. 服务端加 LRU/Redis 缓存：
   - key：`layerId:z:x:y:tileVersion`
   - value：MVT Buffer

### 建议改动文件

- `apps/api/src/layers/layers.controller.ts`
- `apps/api/src/layers/layers.service.ts`
- `apps/api/src/postgis/postgis.repository.ts`
- `apps/api/src/types.ts`
- `apps/web/src/types/gis.ts`
- `apps/web/src/composables/useWebGisWorkspace.ts`
- `apps/web/src/composables/useOpenLayersEditor.ts`

### 验收标准

- 连续请求同一瓦片时命中缓存。
- 编辑保存后对应图层 `tileVersion` 变化。
- 前端刷新后能请求新版瓦片。
- 不编辑时平移缩放不反复触发同一瓦片的 PostGIS 实时计算。

## 阶段 3：脏瓦片精确刷新

### 目标

编辑后只刷新受影响瓦片，而不是刷新整个图层。

### 需要实现

1. 服务端保存前读取旧 geometry bbox。
2. 保存后读取新 geometry bbox。
3. 合并 old/new bbox。
4. 按当前可用 zoom 范围计算受影响 tile 坐标。
5. 返回：

```json
{
  "feature": {},
  "tileVersion": 12,
  "dirtyTiles": [
    { "z": 10, "x": 843, "y": 421 },
    { "z": 11, "x": 1686, "y": 842 }
  ]
}
```

6. 前端拿到 `dirtyTiles` 后：
   - 如果当前视野包含这些 tile，只刷新相关 source。
   - 如果 OpenLayers 无法单 tile 精准刷新，先退化为刷新图层 source。
7. 删除要素时也走 old bbox。
8. 新增要素时只走 new bbox。

### 建议改动文件

- `apps/api/src/postgis/postgis.repository.ts`
- `apps/api/src/layers/layers.service.ts`
- `apps/api/src/types.ts`
- `apps/web/src/composables/useWebGisWorkspace.ts`
- `apps/web/src/composables/useOpenLayersEditor.ts`

### 验收标准

- 小范围编辑不会导致全图层所有瓦片失效。
- 保存后当前视野内相关瓦片能刷新。
- 当前视野外瓦片不立即请求，等用户移动过去时按新版缓存读取。

## 阶段 4：多尺度概化表

### 目标

90w+ 原始表不直接承担所有 zoom 的瓦片生成。

### 需要实现

1. 为大表建立概化/聚合层：
   - 面：`ST_SimplifyPreserveTopology`，按面积阈值过滤。
   - 线：按长度、等级、zoom 容差抽稀。
   - 点：聚合点、网格、H3/quadbin。
2. 服务端按 zoom 选择数据源：
   - z <= 6：聚合/边界/简化表。
   - 7 <= z <= 12：中等概化表。
   - z > 12：原始表。
3. 图层元数据增加概化配置：

```ts
type LayerScaleSource = {
  minZoom: number;
  maxZoom: number;
  schema: string;
  table: string;
  geometryColumn: string;
};
```

4. 查询、选择、编辑仍然走原始表。
5. 浏览 MVT 才走概化表。

### 建议改动文件

- `apps/api/src/postgis/postgis.repository.ts`
- `apps/api/src/layers/layers.repository.ts`
- `apps/api/src/types.ts`
- 后续可新增迁移或管理脚本：`scripts/`

### 验收标准

- 小比例尺不再从原始 90w 表生成瓦片。
- zoom 越大细节越完整。
- 范围选择、点击查询、编辑结果仍然基于原始表。

## 阶段 5：离线瓦片发布

### 目标

对稳定或半稳定大数据，建立类似 GIS 发布服务的瓦片生产链路。

### 可选路线

1. PMTiles：
   - 适合静态文件分发。
   - 前端可通过 PMTiles 协议读取。
   - 可放对象存储/CDN。
2. MBTiles：
   - 适合服务端读取分发。
   - 可以用 SQLite 存储瓦片。
3. 预生成 MVT 目录：
   - 实现简单。
   - 文件数多时需要对象存储或合理分片。

### 编辑后的更新策略

- 小改动：继续用编辑覆盖层即时显示，后台异步重算受影响瓦片。
- 批量编辑：标记图层瓦片版本，后台队列重建。
- 大规模数据更新：重新发布瓦片包。

### 建议新增模块

- `apps/api/src/tile-cache/`
- `apps/api/src/tile-jobs/`
- `scripts/generate-tiles.mjs`
- `scripts/build-simplified-layers.mjs`

### 验收标准

- 大静态图层浏览不依赖 PostGIS 实时 MVT。
- 编辑保存后仍然能通过覆盖层立即显示。
- 后台重建后刷新页面能看到瓦片中的新结果。

## 阶段 6：并发编辑与冲突处理

### 目标

避免多人编辑同一要素时出现静默覆盖。

### 需要实现

1. 图层扫描时识别版本字段：
   - `updated_at`
   - `xmin`
   - 自定义 revision 字段
2. 读取要素时返回版本。
3. 保存时带上版本。
4. 后端更新时校验版本。
5. 冲突时返回 409，让前端提示：
   - 覆盖保存
   - 放弃本地修改
   - 重新加载服务端版本

### 建议改动文件

- `apps/api/src/postgis/postgis.repository.ts`
- `apps/api/src/layers/layers.controller.ts`
- `apps/web/src/composables/useWebGisWorkspace.ts`
- `apps/web/src/components/workbench/EditInspector.vue`

### 验收标准

- 两个客户端同时编辑同一要素，后保存者不会静默覆盖先保存者。
- 冲突提示能展示当前本地草稿和服务端版本。

## 推荐实施顺序

1. 阶段 0：状态边界梳理。
2. 阶段 1：编辑后即时反馈。
3. 阶段 2：图层版本与瓦片缓存。
4. 阶段 3：脏瓦片精确刷新。
5. 阶段 4：多尺度概化表。
6. 阶段 5：离线瓦片发布。
7. 阶段 6：并发编辑与冲突处理。

## 注意事项

- 编辑永远以原始表为准，不以概化表或瓦片为准。
- 范围查询、点击查询、属性表查询应继续走 PostGIS 原始表。
- 浏览层可以延迟一致，编辑覆盖层必须立即一致。
- 缓存可以提升浏览速度，但不能让保存后的当前编辑对象显示旧状态。
- 大表必须依赖空间索引，所有空间查询都要先 bbox 过滤再精确判断。
