# 后端 NestJS 重构进度

## 本次目标

根据用户要求“后端也要修改”并结合 `nestjs-best-practices` 技能，将原先 Fastify 单入口和函数式后端整理为 NestJS 风格的模块化架构，落实 feature module、constructor injection、DTO + ValidationPipe、ConfigModule、Repository Pattern、服务层异常和测试模块等实践，同时保持既有 WebQGIS API 路由兼容。

## 做了哪些

- 将后端入口从 `src/server.ts` 迁移为 NestJS `src/main.ts`：
  - 使用 `NestFactory` + `FastifyAdapter`。
  - 保留全局前缀 `/api`，继续兼容前端现有接口。
  - 启用 CORS。
  - 启用全局 `ValidationPipe`，包含 whitelist、forbidNonWhitelisted、transform。
  - 启用 `ClassSerializerInterceptor`。
  - 启用 shutdown hooks。
- 新增 `AppModule` 和配置模块：
  - `ConfigModule.forRoot` 全局配置。
  - `app.config.ts` 管理 API host、port、环境。
  - `validation.ts` 使用 zod 校验环境变量。
- 按功能拆分模块：
  - `HealthModule`：提供 `GET /api/health`。
  - `DatasourcesModule`：数据源测试、保存、列表、扫描。
  - `LayersModule`：图层列表、要素读写删、MVT 瓦片。
  - `PostgisModule`：PostGIS 查询与连接池 repository。
  - `StorageModule`：JSON 文件读写。
  - `LayerRegistryModule`：图层注册表仓库，避免数据源模块和图层模块循环依赖。
- 数据源模块重构：
  - 新增 `CreateDatasourceDto`、`DatasourceParamDto`、`DatasourceResponseDto`。
  - 使用 DTO 做请求体验证和响应脱敏。
  - `DatasourcesService` 负责业务流程，controller 保持薄。
  - `DatasourcesRepository` 管理本地数据源配置存储。
- 图层模块重构：
  - 新增 `LayerParamDto`、`FeatureParamDto`、`TileParamDto`、`FeaturePayloadDto`。
  - `LayersService` 负责图层和数据源查找、调用 PostGIS repository。
  - `LayersController` 只处理路由和 DTO。
  - `LayersRepository` 管理图层注册表。
- PostGIS 重构：
  - 将原 `postgis.ts` 的 SQL 能力迁移到 `PostgisRepository`。
  - 保留空间表扫描、字段扫描、extent 扫描、MVT 输出、要素读取、新增、更新、删除。
  - 使用 NestJS `BadRequestException` 表达不可查询、不可编辑、缺少几何等错误。
  - 实现 `OnApplicationShutdown` 关闭连接池。
- 删除旧文件：
  - `apps/api/src/server.ts`
  - `apps/api/src/postgis.ts`
  - `apps/api/src/store.ts`
- 新增 API Vitest 配置和服务层测试：
  - `vitest.config.ts`
  - `datasources.service.spec.ts`
  - 使用 `@nestjs/testing` 和 mocked repository 验证脱敏、保存前连接测试、扫描后替换图层注册。
- 已执行验证：
  - `npm install` 成功。
  - `npm run typecheck` 成功。
  - `npm --workspace apps/api run test` 成功，1 个测试文件、3 个测试通过。
  - `npm run build` 成功。
  - 启动 `apps/api/dist/main.js` 后访问 `GET http://localhost:4100/api/health` 成功。

## 没做哪些

- 未接入真实 PostGIS 示例库做端到端扫描、MVT、CRUD 联调。
- 未引入完整认证、授权、角色守卫、限流和审计。
- 未使用数据库迁移工具，当前运行时配置仍保存到本地 JSON 文件。
- 未对 PostGIS SQL 做集成测试，当前测试集中在服务层业务编排。
- 未实现统一异常过滤器和结构化日志输出，仍使用 Nest/Fastify 默认异常响应和日志。
- 未对 datasource password 做加密存储，仍沿用本地 MVP 存储方式。

## 哪里有 bug 或风险

- PostGIS 相关 SQL 尚未在真实库上验证，复杂 schema、视图、多 geometry 字段、权限边界仍可能需要修正。
- `FeaturePayloadDto.geometry` 目前只能验证存在性，尚未校验 GeoJSON 结构合法性。
- 本地 JSON 存储不适合生产，多进程并发写入可能产生竞争。
- 当前 API 响应 DTO 只覆盖数据源脱敏，图层响应仍直接返回注册表对象，后续如加入敏感字段需补响应 DTO。
- `ClassSerializerInterceptor` 已启用，但部分返回值仍是 plain object，后续需要统一响应 DTO 策略。

## 建议后续做什么

- 增加 Docker Compose 示例 PostGIS，提供点、线、面样例表，用于自动化集成测试。
- 为 `PostgisRepository` 增加基于测试库的扫描、MVT、CRUD 集成测试。
- 增加全局异常过滤器，统一返回错误码、中文提示、请求路径和时间戳。
- 将本地 JSON 注册表迁移为数据库表，并对数据源密码加密。
- 增加认证和图层级权限守卫，避免所有接口裸露。
- 为 GeoJSON payload 增加结构化校验和保存前 `ST_IsValid` 友好错误提示。
