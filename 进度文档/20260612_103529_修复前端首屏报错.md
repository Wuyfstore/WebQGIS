# 修复前端首屏报错进度文档

## 本次目标

排查并修复项目启动后前端刚进入页面就报错的问题，保持上一轮 QGIS 风格前端重构效果不回退，并补充回归测试防止同类首屏渲染错误再次出现。

## 做了哪些

- 检查当前工作区状态，确认除旧进度文档外没有其它未提交业务改动。
- 运行 `npm run typecheck`，确认问题不是 TypeScript 类型错误。
- 排查 `WebGisWorkbench.vue` 中模板直接访问 `workspace.xxx.value`、`editor.xxx.value` 的高风险写法。
- 将工作台模板所需的 ref/computed 从 `workspace` 和 `editor` 中解构到 `<script setup>` 顶层。
- 增加 `activeLayerLabel`、`activeLayerEditStatus` 等 computed，让模板读取普通顶层绑定，避免嵌套 ref 自动解包后再访问 `.value` 引起运行时错误。
- 保留 QGIS 风格菜单栏、工具栏、停靠面板、地图画布、属性面板等界面结构。
- 新增 `WebGisWorkbench.test.ts`，mock OpenLayers 编辑器组合函数，验证首屏挂载不崩溃。

## 没做哪些

- 没有修改后端 API 或数据库相关逻辑。
- 没有调整 QGIS 风格视觉设计，只修复首屏运行时风险。
- 没有接入真实 PostGIS 数据源做完整端到端编辑链路验证。
- 没有解决 Vite chunk 超过 500KB 的既有提示。

## 哪里有 bug

- 已定位并修复的 bug：`WebGisWorkbench.vue` 模板中大量通过 `workspace.xxx.value` / `editor.xxx.value` 访问嵌套 ref。Vue 模板会自动解包顶层 ref，但对嵌套对象里的 ref 访问容易在运行时出现不符合预期的 `.value` 读取，导致首屏渲染报错。
- 当前验证未发现新的阻塞性 bug。
- 已知非阻塞问题：`npm run build` 仍提示前端 chunk 超过 500KB。

## 建议后续做什么

- 将工作台 shell 的状态派生继续收敛到 computed，减少模板中的复杂表达式。
- 为数据源扫描、图层选择、样式修改、绘制保存等关键交互增加组件级或端到端测试。
- 后续如果 Windows 自动化环境恢复，可补一次真实浏览器截图验证。
- 对 OpenLayers 相关代码做分包或懒加载，降低构建 chunk 体积。
