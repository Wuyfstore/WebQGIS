# Chrome 验证并移除 UnoCSS 焦点环

## 本次目标

- 使用 Chrome 插件直接检查 WebQGIS 页面中属性表 Tab 点击后仍出现蓝色边框的问题。
- 定位真实浏览器里产生蓝色高亮的 computed style 和 CSS 规则。
- 移除 UnoCSS 生成的蓝色焦点环，确保点击属性表 Tab 后不再出现蓝色边框。

## 做了哪些

- 使用 Chrome 插件接管当前 `http://localhost:5173/` 的 WebQGIS 标签页。
- 在 Chrome 中检查 `.attribute-table__tab` 的 computed style 和命中的 CSS 规则。
- 确认真实来源不是 `border` 或 `outline`，而是 UnoCSS 生成的 `.focus-ring:focus` 规则通过 `box-shadow` 和 `--un-ring-*` 变量产生焦点环。
- 在 `apps/web/src/styles.css` 中为 `.focus-ring:focus` 增加 Uno ring 变量覆盖：
  - `--un-ring-color: transparent`
  - `--un-ring-offset-width: 0px`
  - `--un-ring-width: 0px`
  - `--un-ring-offset-shadow: 0 0 #0000`
  - `--un-ring-shadow: 0 0 #0000`
- 重新加载 Chrome 页面，打开属性表并点击“属性数据”Tab，确认当前 focused Tab 的 computed `boxShadow` 已变为透明，`--un-ring-color` 为 `transparent`，`--un-ring-width` 为 `0px`。
- 执行并通过 `npm --workspace apps/web run test`。
- 执行并通过 `npm --workspace apps/web run typecheck`。

## 没做哪些

- 没有修改属性表 Tab 的业务逻辑、切换逻辑或 ARIA 属性。
- 没有修改顶部菜单交互逻辑。
- 没有处理工作区中已有的无关删除/修改。

## 哪里有 bug

- 本次修复前的问题是：UnoCSS 的 `.focus-ring:focus` 仍会通过 `box-shadow` 生成焦点环，即使 `outline` 已改成灰色或被禁用，点击后仍可能看到蓝色边框。
- 本次修复后，Chrome 实测 focused 属性表 Tab 的 Uno ring 已被清零，未发现新增功能 bug。
- 如果用户浏览器仍显示旧蓝框，建议强制刷新页面或重启 Vite dev server，避免旧 `virtual:uno.css` 缓存残留。

## 建议后续做什么

- 将 `focus-ring` 的职责统一到一处，避免 UnoCSS shortcut 和全局 CSS 同时定义焦点效果。
- 后续可以补一个浏览器级视觉回归用例，专门覆盖属性表 Tab 点击后的焦点环颜色。
