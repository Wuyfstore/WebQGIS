# 20260615_235438_QGIS式编辑流程

## 本次目标

- 将 WebQGIS 的编辑流程调整为更接近 QGIS：图层可编辑不等于已经开启编辑。
- 默认进入工作台或切换图层后编辑模式关闭，用户必须显式点击“开启编辑”后才能绘制、保存、删除、清空草稿或修改属性。
- 使用 Chrome 插件核对当前 WebQGIS 真实界面行为。
- 完成测试、类型检查、Git 提交并推送到 main 分支。

## 做了哪些

- 在工作台增加当前正在编辑的图层状态 `editingLayerId`，用 `isEditingActiveLayer` 派生当前图层是否处于编辑模式。
- 顶部工具栏新增“编辑”切换按钮，可在“开启当前图层编辑 / 关闭当前图层编辑”之间切换。
- 项目、编辑、矢量菜单中的编辑类命令增加编辑模式门禁，未开启编辑时不可直接绘制、保存、删除或撤销草稿。
- 地图浮动编辑工具栏从“编辑图层”改为“活动图层”，新增“开启编辑 / 关闭编辑”按钮。
- 未开启编辑时禁用绘制类型选择、绘制、保存、删除、清空按钮，并显示“先开启编辑，再绘制、修改或保存要素”。
- 属性表单在未开启编辑时禁用输入框，并提示“开启当前图层编辑后可修改属性”。
- 切换活动图层时自动关闭编辑模式、清空草稿，并显示“已切换图层，编辑模式已关闭”。
- 增加硬保护：即使命令被误触发，保存、删除、绘制、新建草稿也会先检查是否已开启编辑。
- 更新 `WebGisWorkbench.test.ts`，覆盖默认关闭编辑、开启编辑后可绘制、切换图层后自动关闭编辑等行为。
- 使用 Chrome 插件验证：
  - 初始状态显示“编辑: 关闭”。
  - 地图工具条显示“开启编辑”。
  - 绘制、保存、删除、清空和属性输入在未开启编辑时禁用。
  - 点击“开启编辑”后状态变为“编辑中”，属性输入解锁，绘制按钮可用。
  - 切换图层后编辑模式关闭，属性输入重新禁用。
- 执行 `npm --workspace apps/web run test`，3 个测试文件、23 个测试通过。
- 执行 `npm --workspace apps/web run typecheck`，类型检查通过。

## 没做哪些

- 没有修改后端接口、PostGIS 保存逻辑或 MVT 加载逻辑。
- 没有改变图层是否具备可编辑能力的判定规则。
- 没有实现 QGIS 完整的编辑会话、撤销栈、保存前确认或未保存修改提醒。
- 没有处理与本次无关的工作区改动：`README.md` 删除、`设计稿/*v2*` 删除、旧进度文档修改。

## 哪里有 bug

- 本次测试和 Chrome 验证未发现新增阻断 bug。
- 当前仍存在产品层面的后续风险：关闭编辑或切换图层时现在会清空草稿，但还没有像 QGIS 一样在存在未保存修改时弹出“保存 / 丢弃 / 取消”的确认流程。
- 属性输入解锁后是否应要求先选择要素再允许编辑字段，仍取决于现有业务流程，本次未进一步收紧。

## 建议后续做什么

- 增加“有未保存草稿时关闭编辑 / 切换图层”的确认弹窗，模拟 QGIS 的保存编辑提醒。
- 给顶部“编辑”按钮和地图工具条“开启编辑”按钮统一状态文案，减少用户理解成本。
- 增加端到端测试，覆盖真实浏览器中开启编辑、绘制草稿、保存、切换图层的完整链路。
- 后续可把编辑会话状态抽成 composable，降低 `WebGisWorkbench.vue` 的编排复杂度。
