# 修复空间表拖动添加图层无效

## 本次目标

- 排查并修复数据源展开空间表后，拖动空间表行到地图或图层面板无法添加图层的问题。
- 保持双击添加图层、拖到地图添加、拖到图层面板添加三条入口一致可用。

## 做了哪些

- 排查拖拽事件链，定位到 `dragover` 阶段使用 `dataTransfer.getData()` 判断拖拽来源会导致真实浏览器无法触发 `drop`。
- 在真实浏览器中，`dragover` 阶段经常只能读取 `dataTransfer.types`，不能可靠读取 `getData()` 内容；旧逻辑因此没有执行 `preventDefault()`，浏览器会直接拦截后续 `drop`。
- 修改 `MapCanvas.vue`：`dragover` 改为通过 `dataTransfer.types` 判断空间表拖拽 payload，允许 drop；`drop` 阶段继续读取 layer id 并发出 `layerDrop`。
- 修改 `LayerPanel.vue`：同样用 `dataTransfer.types` 判断拖拽 payload，保证拖到图层面板也能触发 drop。
- 更新 `WebGisWorkbench.test.ts`：补充 `dragover` 中 `types` 有值但 `getData()` 为空的回归场景，并断言地图与图层面板都会执行 `preventDefault()`。
- 用 Chrome 检查当前页面：展开连接后 4 个空间表均为 `draggable=true`，地图区域和图层面板 drop target 均存在。

## 没做哪些

- 未修改后端接口、MVT 渲染、空间表扫描逻辑。
- 未改变双击添加图层逻辑。
- 未新增图层排序、图层移除、工程保存等功能。

## 哪里有 bug

- 本次修复前的 bug：真实浏览器拖拽空间表行时，drop 目标没有正确允许 drop，导致拖到地图或图层面板没有效果。
- 本次修复后未发现新的功能性 bug。
- Chrome 插件的脚本环境不允许构造 `Event` / `DataTransfer` 进行注入式 drop 验证，因此运行时验证以 DOM 状态检查为主；关键失败条件已通过 Vitest 回归测试覆盖。

## 建议后续做什么

- 增加 Playwright E2E 测试，使用真实浏览器 drag/drop 验证空间表拖到地图和图层面板的完整链路。
- 可考虑抽取共享拖放 helper，减少 `MapCanvas` 与 `LayerPanel` 的重复判断逻辑。
- 给拖放目标增加更明显的 hover/drop 反馈，便于用户确认当前位置可放置。
