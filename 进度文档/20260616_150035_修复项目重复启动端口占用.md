# 修复项目重复启动端口占用进度文档

## 本次目标

- 排查当前项目启动失败原因并处理。
- 保证根目录 `npm run dev` 能在已有旧 WebQGIS dev 进程残留时重新启动。
- 完成验证后生成进度文档，并提交推送到 `main` 分支。

## 做了哪些

- 复现并排查启动问题：
  - Web 构建、Web 类型检查、API 类型检查、API 构建均可通过，说明不是静态编译错误。
  - 通过日志方式启动 API，捕获到明确错误：`Error: listen EADDRINUSE: address already in use 0.0.0.0:4100`。
  - 检查系统端口后发现多个旧 WebQGIS dev 进程残留：API 占用 `4100`，多个 Vite 进程占用 `5173/5174/5175`。
- 新增 `scripts/dev-clean-start.mjs`：
  - 启动前扫描当前仓库路径下的旧 `npm-run-all`、`vite`、`tsx watch`、workspace dev 进程。
  - 只清理属于当前 WebQGIS 仓库的旧 dev 进程及其子进程。
  - 清理后重新启动原来的 `npm-run-all --parallel dev:*`。
  - 兼容 Windows 和非 Windows 环境。
- 修改根目录 `package.json`：
  - 将 `dev` 从 `npm-run-all --parallel dev:*` 改为 `node scripts/dev-clean-start.mjs`。
  - 保留 `dev:api`、`dev:web`，方便单独启动服务。
- 修复过程中发现 Windows 上直接 `spawn("npm-run-all.cmd")` 会报 `spawn EINVAL`，已改为 Windows 下通过 `cmd.exe /d /s /c "npm-run-all --parallel dev:*"` 启动。
- 验证新的 `npm run dev`：
  - 清理旧进程后只监听 `4100` 和 `5173`。
  - `http://127.0.0.1:4100/api/health` 返回 `{"ok":true,"service":"webqgis-api",...}`。
  - `http://127.0.0.1:5173` 返回 `200`。

## 没做哪些

- 未修改 WebQGIS 前端业务逻辑。
- 未修改 API 业务接口。
- 未变更数据库连接、PostGIS 扫描、图层加载或地图渲染逻辑。
- 未处理工作区已有的无关未提交改动，包括 README 删除、设计稿删除和旧进度文档修改。

## 哪里有 bug

- 原启动失败根因：旧的 WebQGIS API dev 进程占用 `0.0.0.0:4100`，导致新 API 启动报 `EADDRINUSE`。
- 关联问题：重复启动时 Vite 会自动漂移到 `5174/5175`，导致用户看到的访问地址混乱。
- 修复过程中的脚本问题：Windows 直接 spawn `npm-run-all.cmd` 报 `spawn EINVAL`，已改为通过 `cmd.exe` 启动。
- 当前未发现修复后仍存在的启动阻断 bug。

## 建议后续做什么

- 后续可增加 `npm run stop:dev`，单独清理当前仓库 dev 进程。
- 可在 API 启动失败时输出更友好的端口占用提示，包括占用进程 PID。
- 可让 Vite 固定 `strictPort: true`，避免端口自动漂移造成访问地址混乱；本次先通过启动前清理旧进程解决根因。
- 可在 README 或开发文档补充 Windows 下重复启动的处理方式。
