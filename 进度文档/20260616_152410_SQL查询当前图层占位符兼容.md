# SQL 查询当前图层占位符兼容

## 本次目标

- 处理属性表 SQL 查询中输入 `select * from {public.china_2025_city}` 返回 400 的问题。
- 明确并兼容当前图层的 `{schema.table}` 占位符写法，同时保留推荐的 `{layer}` 写法。
- 不放开跨图层查询、不改变只读 SELECT 和 LIMIT 安全限制。

## 做了哪些

- 后端 SQL 安全构造逻辑从只替换 `{layer}` 调整为统一解析图层占位符。
- 支持当前图层对应的 `{schema.table}` 占位符，例如当前图层为 `public.china_2025_city` 时可使用 `select * from {public.china_2025_city}`。
- 对非当前图层的 `{schema.table}` 占位符返回更明确的错误信息，避免误以为可以任意查询其他表。
- 补充 Vitest 单测，覆盖当前图层 `{schema.table}` 成功和错误图层占位符失败。
- 更新属性表 SQL 查询说明，提示推荐使用 `{layer}`，也支持当前表名占位符 `{schema.table}`。

## 没做哪些

- 没有允许跨图层或跨表 SQL 查询。
- 没有允许 UPDATE、DELETE、DDL、多语句、注释等非只读 SQL。
- 没有调整属性表 SQL 查询结果展示样式。
- 没有新增后端接口或修改数据库连接流程。

## 哪里有 bug

- 本次修复前，后端只识别固定 `{layer}` 占位符，用户输入 `{public.china_2025_city}` 会因为未替换且不匹配当前表校验而返回 400。
- 当前未发现新增 bug。

## 建议后续做什么

- 在 SQL 输入框旁增加一个“插入当前图层”按钮，直接填入 `{layer}`，减少用户手写占位符的机会。
- 将后端 SQL 400 错误在前端以更醒目的方式展示，区分语法错误、占位符错误和权限/安全限制错误。
- 后续如需支持更复杂查询，应设计明确的只读查询沙箱和表白名单，而不是直接放开任意 SQL。
