# SQL 查询占位符传参说明

## 本次目标

- 说明 `{"sql":"select * from {china_2025_city}","limit":100}` 为什么在属性表 SQL 查询接口中返回 400。
- 明确正确传参方式，避免继续使用缺少 schema 的非法占位符。

## 做了哪些

- 确认当前 SQL 查询接口是按图层 ID 作用域执行的 `/api/layers/:id/query`。
- 明确 `{china_2025_city}` 不是支持的占位符格式，因此会触发后端安全校验并返回 400。
- 给出推荐写法：`select * from {layer}`。
- 说明如果当前图层确实是 `public.china_2025_city`，也可以使用完整当前表名占位符：`select * from {public.china_2025_city}`。

## 没做哪些

- 本次没有修改代码。
- 没有放开 `{table}` 这种缺少 schema 的占位符。
- 没有允许跨图层或跨表查询。
- 没有调整后端只读 SELECT、安全校验和 LIMIT 逻辑。

## 哪里有 bug

- 用户传参中的 `{china_2025_city}` 缺少 schema，也不是固定占位符 `{layer}`，因此不是合法占位符。
- 当前没有发现新增 bug。

## 建议后续做什么

- 前端 SQL 输入区可以增加示例按钮或占位符插入按钮，优先插入 `select * from {layer}`。
- 后端错误信息可以继续优化为更接近用户输入场景的中文提示。
- 如果未来需要支持 `{table}` 简写，应先明确数据源和 schema 解析规则，避免同名表造成歧义。
