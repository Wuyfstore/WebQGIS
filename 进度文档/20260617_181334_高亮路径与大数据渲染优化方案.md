# 高亮路径与大数据渲染优化方案

## 本次目标

- 回答点击高亮和范围查询高亮是否走同一个渲染途径。
- 判断范围查询返回结果高亮后，前一个点击高亮是否应该移除。
- 给出 90w+ 数据量下，矢量瓦片实时绘制之外的优化方案。
- 本次按用户要求“给出方案即可”，不做业务代码改动。

## 做了哪些

- 核对前端 `useOpenLayersEditor` 的高亮相关代码：
  - 点击选择命中瓦片要素后，通过 `readFeature()` 回源读取单个 GeoJSON，再调用 `loadEditableFeature()`。
  - `loadEditableFeature()` 会清空 `selectedFeatureSource`，再把要素加入 `editSource`。
  - 点击选择还可能通过 workspace 的 `selectedFeatureId` 影响业务 MVT 图层样式函数，让原瓦片要素本身变成橙色。
  - 范围选择通过 `/features/select` 返回的 GeoJSON features 调用 `loadSelectedFeatures()`，写入 `selectedFeatureSource`。
  - 范围选择 fallback 使用 `loadRenderedSelectionFeatures()`，同样写入 `selectedFeatureSource`。
- 核对服务端 PostGIS 路径：
  - 范围选择接口使用 `ST_Intersects` 返回 `id, feature`。
  - 矢量瓦片接口使用 `ST_AsMVTGeom` 和 `ST_AsMVT` 实时生成 MVT。
- 给出结论：
  - 当前点击高亮和范围查询高亮不是完全同一路径。
  - 点击选择使用 `editSource/editLayer`，范围查询高亮使用 `selectedFeatureSource/selectedFeatureLayer`。
  - 范围查询结果高亮后，前一个点击高亮或编辑草稿如果不再处于编辑场景，应该被移除，否则两个橙色图层叠加会让用户误判。

## 方案

### 一、高亮路径建议

当前路径：

- 点击选择：`handleFeaturePick()` -> `readFeature()` -> `loadEditableFeature()` -> `editSource/editLayer`；同时业务 MVT 图层样式可能根据 `selectedFeatureId` 把瓦片要素染成橙色。
- 范围选择：`selectFeatureInGeometry()` -> `loadSelectedFeatures()` 或 `loadRenderedSelectionFeatures()` -> `selectedFeatureSource/selectedFeatureLayer`。

建议调整为：

1. 区分“查看高亮”和“编辑草稿”两个概念。
   - 点击选择如果只是选择/识别，应进入统一的 `selectedFeatureSource`。
   - 只有用户明确进入节点编辑、绘制、保存编辑时，才把要素放入 `editSource`。
2. 范围选择成功写入 `selectedFeatureSource` 前，应清空 `editSource` 中的非编辑草稿。
3. 范围选择成功前还应调用 workspace 的清理方法，清掉 `selectedFeatureId/selectedProperties`，避免业务 MVT 图层继续保留前一个点击要素的橙色样式。
4. 如果当前确实处于编辑模式，要避免范围选择静默清掉未保存草稿，应先提示或禁止切换选择范围。
5. 最小改法：
   - 在范围选择成功前调用一个 `clearSelectionVisuals({ keepDraft: false })`。
   - 该方法清 `selectedFeatureSource`、`editSource`、`draftGeometry`、`selectedFeatureId/selectedProperties`，然后范围结果只写入 `selectedFeatureSource`。
   - 如果要保留编辑草稿，则范围选择入口应先提示用户保存或取消编辑。

### 二、90w+ 数据渲染优化短期方案

1. 保留 MVT，但禁止“实时无缓存裸算”。
   - 对 `/tile/{z}/{x}/{y}.mvt` 增加 HTTP 缓存头、ETag、内存/LRU 缓存或 Redis 缓存。
   - 图层数据有更新时按图层版本号刷新缓存 key。
2. 按 zoom 分级限制。
   - 小比例尺只返回概化面、聚合点、边界骨架或热力/栅格摘要。
   - 中大比例尺才返回真实要素。
3. 建立并确认空间索引。
   - geometry GiST/SP-GiST 索引必须存在。
   - `ST_Intersects` 前保留 bbox `&&` 过滤。
4. 预生成低 zoom 瓦片。
   - 低 zoom 被访问频率高，实时算最浪费。
   - 可对 z0-z8 或 z0-z10 做离线预切片。

### 三、90w+ 数据渲染优化中期方案

1. 建立多尺度概化表。
   - 原始表：用于查询、编辑、高 zoom。
   - `layer_simplified_z6/z8/z10`：用于小比例尺展示。
   - 使用 `ST_SimplifyPreserveTopology` 或按业务容差生成。
2. 引入瓦片物化表或 MBTiles/PMTiles。
   - 稳定底图型数据建议离线生成 PMTiles/MBTiles。
   - 前端直接加载静态瓦片，服务端只负责权限、元数据、编辑和查询。
3. 查询和渲染分离。
   - 地图展示走概化/缓存瓦片。
   - 范围选择、属性查询走 PostGIS 原始表和索引。
   - 高亮结果只返回命中要素的轻量 GeoJSON 或高亮专用 MVT。

### 四、90w+ 数据渲染优化长期方案

1. 架构上拆成“浏览层、查询层、编辑层”。
   - 浏览层：缓存 MVT/PMTiles/栅格化切片。
   - 查询层：PostGIS 原始表，负责精确空间查询。
   - 编辑层：只加载当前编辑要素和邻近上下文。
2. 对超大点/线数据使用聚合或服务端分桶。
   - 点：聚合点、H3/quadbin 网格、热力图。
   - 线：按 zoom 抽稀、按长度/等级过滤。
   - 面：按行政等级、面积阈值、概化表分层。
3. 对不可编辑的大底图考虑栅格化。
   - 如果只是背景浏览，不需要每个要素前端可交互，栅格瓦片会比 MVT 更稳。
   - 保留查询 API 实现点击/范围查询即可。

## 哪里有 bug

- 现有设计风险：点击选择和范围选择使用不同高亮 source，范围结果出现后，点击高亮对应的 `editSource` 可能仍然存在，视觉上会出现旧高亮和新高亮叠加。
- 现有状态风险：范围选择没有清 workspace 的 `selectedFeatureId` 时，业务 MVT 图层可能继续把上一次点击要素渲染为橙色。
- 现有性能风险：90w+ 数据如果依赖 PostGIS 每次实时生成矢量瓦片，在小比例尺、频繁平移缩放、多用户并发时容易出现 CPU 和数据库压力。
- 现有体验风险：如果为了清理旧点击高亮直接清空 `editSource`，可能误删未保存编辑草稿，因此需要先明确“选择高亮”和“编辑草稿”的状态边界。

## 建议后续做什么

- 优先改造高亮状态模型：选择高亮统一走 `selectedFeatureSource`，编辑草稿独立走 `editSource`。
- 给范围选择成功路径增加“清理旧点击高亮”的明确逻辑，并保护未保存编辑草稿。
- 给 MVT 接口加缓存和图层版本 key。
- 为大表建立多尺度概化表，并让前端按 zoom 选择不同 tileUrl 或服务端按 zoom 自动路由。
- 对稳定大数据图层评估 PMTiles/MBTiles 离线切片。
