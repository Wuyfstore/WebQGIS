# 阶段2tileVersion与瓦片缓存

## 本次目标

按照 `设计文档/大数据矢量编辑性能优化实施方案.md` 的阶段 2，给 PostGIS MVT 浏览层增加图层级 `tileVersion` 和最小可用的服务端瓦片缓存。保存编辑后递增图层版本，前端使用新版 URL 请求瓦片，让浏览层能缓存且能在编辑后失效。

## 做了哪些

- `LayerRegistration` 增加 `tileVersion`，扫描新图层时默认写入 `1`。
- `LayersRepository.findAll()` 对旧 `layers.json` 自动补 `tileVersion: 1`，避免旧注册文件破坏类型和前端逻辑。
- `replaceForDatasource()` 保留已有图层的样式和 `tileVersion`，避免重新扫描数据源导致瓦片版本倒退。
- 新增 `LayersRepository.bumpTileVersion()`，保存、新增、删除要素成功后递增版本并更新时间。
- 新增 `FeatureWriteResult`，`createFeature()` / `updateFeature()` 返回 `{ feature, tileVersion }`。
- 删除要素接口改为返回 `{ tileVersion }`，便于前端同步图层版本。
- `LayersService` 增加内存 MVT Buffer 缓存，缓存 key 为 `layerId:z:x:y:tileVersion`。
- MVT 响应增加长期缓存头：`Cache-Control: public, max-age=31536000, immutable`，并设置 ETag。
- 前端 `LayerRegistration` 和保存响应类型同步增加 `tileVersion`。
- 前端保存成功后兼容两种返回：
  - 新格式 `{ feature, tileVersion }`：更新选中 Feature、草稿 geometry 和图层版本。
  - 旧格式 `Feature`：继续按阶段 1 逻辑工作。
- OpenLayers MVT URL 增加 `?v=tileVersion` 或 `&v=tileVersion`。
- 已挂载的 VectorTileSource 在图层版本变化时更新 URL 并刷新 source。
- 补充 API 和 Web 测试，覆盖版本保留、版本递增、缓存命中、保存返回新格式、旧格式兼容和 URL 版本参数。
- 使用已有子 agent 做了后端、前端只读探查，核对实现落点。

## 没做哪些

- 未实现 Redis、磁盘缓存或 LRU 淘汰；当前是进程内 `Map`，适合阶段 2 的最小可用验证。
- 未实现 dirtyTiles 精确刷新；当前保存后仍按图层级 `tileVersion` 失效。
- 未做真实 PostGIS 写入后的端到端瓦片请求命中统计。
- 未实现服务端按请求参数 `v` 回退历史版本；当前缓存版本以服务端注册表中的当前 `tileVersion` 为准，URL 版本主要用于浏览器/CDN 缓存失效。

## 哪里有 bug / 风险

- 内存缓存没有容量上限，长期运行或大量图层/缩放访问可能增长过大，后续阶段应换成 LRU 或 Redis。
- 多 API 实例部署时，`tileVersion` 写在 `layers.json`，但内存缓存不共享；多实例需要共享缓存或统一失效机制。
- 删除接口从 204 空响应改为 200 JSON 响应，前端已兼容，但如果外部客户端依赖 204 需要同步调整。
- 当前 ETag 使用请求里的 `v` 或 `current`，没有强校验请求版本是否等于服务端版本。

## 建议后续做什么

- 阶段 3 实现 `dirtyTiles`，让编辑后只刷新受影响瓦片，减少整层版本失效。
- 给 MVT 缓存加容量上限、TTL 或 LRU，必要时抽象成可替换的缓存服务。
- 增加真实浏览器或 API E2E，验证保存后 `/tile/...mvt?v=newVersion` 请求能触发新版瓦片。
- 后续在 dirtyTiles 刷新完成后清理阶段 1 的 `coveredFeatureIdsByLayer`。
