# 阶段3dirtyTiles精确刷新

## 本次目标

按照 `设计文档/大数据矢量编辑性能优化实施方案.md` 的阶段 3，为新增、更新、删除要素返回 `dirtyTiles`，让前端保存后具备按受影响瓦片刷新浏览层的响应契约。第一版前端仍退化为刷新整个 OpenLayers source，但接口和状态已接住 dirtyTiles，后续可替换为单瓦片失效。

## 做了哪些

- 新增 `DirtyTile`、`GeometryBbox`、`FeatureDeleteResult`、`FeatureMutationResult` 等类型。
- 新增 `apps/api/src/tiles/dirty-tiles.ts`，提供：
  - `mergeBboxes()`
  - `dirtyTilesForBbox()`
- `dirtyTilesForBbox()` 使用 WGS84 bbox 转 WebMercator slippy tile 坐标，默认覆盖 z0-z14，并对超大范围做上限保护。
- `PostgisRepository.createFeature()` 现在返回保存后的 Feature 和新 geometry bbox。
- `PostgisRepository.updateFeature()` 先读取旧 Feature，再写入并读取新 Feature，返回 old/new bbox。
- `PostgisRepository.deleteFeature()` 删除前读取旧 Feature，返回 old bbox。
- 新增 `geometryBbox()` 递归计算 GeoJSON geometry bbox。
- `LayersService.createFeature()` / `updateFeature()` / `deleteFeature()` 根据 mutation bbox 计算 dirtyTiles，并随 `tileVersion` 一起返回。
- 前端 `FeatureWriteResult` 和新增 `FeatureDeleteResult` 接收 dirtyTiles。
- `useWebGisWorkspace()` 保存/删除后记录最近一次 dirtyTiles，并提供 `consumeLastDirtyTiles()` 给工作台消费。
- `WebGisWorkbench.vue` 保存/删除后根据 dirtyTiles 进入刷新协调 helper；当前第一版仍调用 `editor.refreshLayer(layerId)` 作为整 source 退化刷新。
- 补充 API 测试：bbox 合并、bbox 转 tile、超大范围保护、GeoJSON bbox、写入响应 dirtyTiles。
- 补充 Web 测试：保存响应带 dirtyTiles 后仍即时覆盖并刷新；删除响应带 dirtyTiles 后刷新图层并更新 tileVersion。
- 使用已有子 agent 做了后端和前端只读探查，核对实现边界。

## 没做哪些

- 未实现 OpenLayers 单瓦片级失效；当前收到 dirtyTiles 后仍刷新整个图层 source。
- 未把 bbox SQL 下推到 PostGIS 直接返回 3857 bbox；当前基于服务端读取到的 4326 GeoJSON 计算 bbox，再转换为 WebMercator tile。
- 未为 dirtyTiles 增加可配置 zoom 范围；当前默认 z0-z14。
- 未实现 dirtyTiles 与阶段 1 `coveredFeatureIdsByLayer` 的自动清理联动。

## 哪里有 bug / 风险

- 对跨大范围面要素，dirtyTiles 会因为上限保护跳过高 zoom 的大量瓦片，避免返回爆炸，但也意味着后续局部刷新需要有整层刷新兜底。
- 属性-only 更新会返回同一旧/新 bbox，这可以刷新 MVT 属性，但会比纯几何变更稍宽。
- 当前删除不存在要素时仍可能返回空 dirtyTiles 并递增版本，后续可进一步强化未命中语义。
- Chrome 冒烟只验证页面加载和 console error，不代表真实 PostGIS 写入后的 dirtyTiles 端到端验证。

## 建议后续做什么

- 阶段 4 前可先把 `refreshLayerAfterMutation()` 内部替换为 OpenLayers 单 tile 失效或视野内 dirty tile 判断。
- 让 dirtyTiles zoom 范围来自图层配置，例如 `minZoom/maxZoom`。
- 在瓦片新版本加载完成后清理 `coveredFeatureIdsByLayer`，让编辑覆盖隐藏状态自动收敛。
- 如果后续性能压力明显，再把 bbox 计算下推到 SQL，直接返回 `ST_Transform(..., 3857)` 的 bbox。
