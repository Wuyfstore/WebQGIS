# 阶段4多尺度概化表

## 本次目标

实现“大数据矢量编辑性能优化实施方案”阶段 4：为大数据矢量浏览层增加多尺度概化表能力，让低缩放级别的 MVT 可以读取更轻量的概化表，同时保持点击读取、范围选择、编辑保存等工作流继续回源原始表，避免把概化数据误用于编辑。

## 做了哪些

- 新增 `LayerScaleSource` 类型，并在前后端 `LayerRegistration` 上增加 `scaleSources` 配置。
- 图层重新扫描覆盖元数据时，保留已配置的 `scaleSources`，避免数据源刷新后丢失多尺度配置。
- `PostgisRepository.getVectorTile()` 按当前 z 级别选择匹配的概化表：
  - 命中 `scaleSources` 时，只切换 MVT 浏览查询使用的 schema/table/geometryColumn/primaryKey。
  - 未命中时继续读取原始表。
  - 多个范围同时命中时优先选择 zoom 范围更窄的配置。
- 保持编辑相关读取仍走原始表：
  - `readFeature()` 不受 `scaleSources` 影响。
  - 范围选择、属性读取、插入、更新、删除仍基于原始图层注册信息。
- 新增概化表辅助脚本：
  - `scripts/build-simplified-layers.mjs`：默认 dry-run 输出建表 SQL，只有显式 `--execute` 才连接数据库执行。
  - `scripts/generate-layer-scale-config.mjs`：根据 zoom 范围生成可写入图层配置的 `scaleSources` JSON。
- 修正概化建表脚本的 SRID 行为：
  - 默认使用 `geometry(MultiPolygon)`，保留源几何自身 SRID，不再硬编码为 4326。
  - 支持显式 `--target-srid 3857` 等参数，用于确实需要固定 typmod 的场景。

## 没做哪些

- 没有做前端 `scaleSources` 可视化配置面板；当前仍需通过配置 JSON 或后续管理入口写入。
- 没有自动判断哪些图层应该生成哪些 zoom 范围的概化表；脚本只提供建表 SQL 和配置生成能力。
- 没有做保存编辑后自动重建概化表；阶段 4 先解决低 zoom 浏览读取轻量表，概化表刷新策略应在后续阶段结合异步任务或定时维护实现。
- 没有改变当前选择高亮、编辑覆盖层、dirtyTiles、tileVersion 的前端刷新机制。

## 哪里有 bug

- 本轮未发现新的阻塞 bug。
- 已特别处理一个潜在问题：概化建表脚本原先如果硬编码输出为 `geometry(MultiPolygon, 4326)`，会对 3857 图层造成坐标系 typmod 错误；现在默认保留源表 SRID，并允许显式指定目标 SRID。
- 当前多尺度概化表假设概化表与原始表存在稳定 id 映射；如果概化表缺失 id、SRID 不一致、几何列名不一致或空间索引缺失，会导致低 zoom MVT 查询不可用或性能下降。
- 当前 MVT 读取概化表时只保留 id 字段作为属性，避免引用概化表不存在的业务字段；如果后续希望低 zoom 仍显示完整属性，需要扩展 `scaleSources` 字段映射。

## 建议后续做什么

- 增加图层配置入口，支持在 UI 或 API 中编辑 `scaleSources`，并校验表是否存在、SRID 是否匹配、id 字段是否可用、空间索引是否存在。
- 设计概化表刷新策略：
  - 小范围编辑后按 dirtyTiles 或 bbox 增量刷新相关概化记录。
  - 大批量编辑后使用后台任务重建概化表。
  - 为概化表增加 `updatedAt` 或版本号，便于排查浏览层延迟一致问题。
- 支持按几何类型生成不同概化 SQL：
  - Polygon/MultiPolygon 使用 `ST_SimplifyPreserveTopology`。
  - LineString/MultiLineString 可使用 `ST_Simplify` 或按道路等级做抽稀。
  - Point/MultiPoint 更适合聚合、网格化或聚类，而不是拓扑概化。
- 在真实 90w+ 数据上压测：
  - 低 zoom 瓦片生成耗时。
  - 概化表大小与索引命中情况。
  - 编辑保存后浏览层最终一致的可感知延迟。
