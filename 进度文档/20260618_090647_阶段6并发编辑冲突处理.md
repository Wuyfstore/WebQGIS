# 阶段6并发编辑冲突处理

## 本次目标

按照 `设计文档/大数据矢量编辑性能优化实施方案.md` 的阶段 6，实现 WebQGIS 矢量要素编辑的基础并发冲突保护：读取要素时返回版本号，保存时携带版本号，服务端发现版本过期时返回 409，前端提示冲突并提供重新加载服务端版本入口。

## 做了哪些

- 后端 GeoJSON Feature 增加顶层 `revision` 字段，使用 PostgreSQL `xmin::text` 作为当前最小可用版本标识。
- 后端要素详情读取 SQL 返回 `revision`，前端选择要素后记录该版本。
- 后端更新要素时，如果请求携带 `revision`，则在 `update ... where` 中追加 `t.xmin::text = revision` 条件。
- 更新命中 0 行时区分不存在与版本冲突：要素仍存在且请求携带版本时抛出 `ConflictException`，由 NestJS 返回 409。
- 保存成功后继续刷新 tileVersion/dirtyTiles；409 冲突时不 bumpTileVersion，避免无效刷新。
- 前端 API 层增加 `ApiError`，保留 HTTP status，便于业务层识别 409。
- 前端编辑状态保存 `selectedFeatureRevision` 和 `featureConflict`，保存请求携带 `revision`。
- 前端遇到 409 时保留当前草稿，不覆盖用户未保存编辑，并在编辑面板显示冲突提示。
- 编辑面板显示当前版本号；冲突时提供“重新加载服务端版本”按钮。
- 重新加载服务端版本后更新编辑器草稿、清除冲突状态，并显示最新版本。
- 补充 API 与 Web 测试，覆盖读取版本、保存携带版本、冲突不刷新瓦片、冲突提示和重新加载流程。
- 使用 Chrome 控制做页面冒烟验证，确认 `http://127.0.0.1:5173/` 标题为 `WebQGIS` 且无控制台 error。

## 没做哪些

- 没有引入显式 `revision` 列、`updated_at` 或乐观锁字段迁移，本阶段先用 PostgreSQL `xmin` 快速落地。
- 没有实现删除要素时的版本冲突保护，本轮只覆盖编辑保存路径。
- 没有实现强制覆盖、差异合并、自动合并或多人实时协同。
- 没有做冲突前后两版几何/属性的可视化 diff。
- 没有改变既有瓦片缓存、离线发布和概化表设计。

## 哪里有 bug

- 本轮自动化测试和 Chrome 冒烟没有发现新增 bug。
- 已知限制：`xmin` 是 PostgreSQL 系统列，适合作为当前阶段的轻量乐观锁，但长期方案建议改成业务显式版本字段，避免 vacuum/freeze、跨表同步和迁移语义带来的不确定性。
- 已知限制：冲突后点击重新加载服务端版本会丢弃当前本地草稿；这是第一版的明确取舍，后续需要在 UI 上提供“保留本地副本/查看差异/覆盖保存”等更完整路径。

## 建议后续做什么

- 为业务表增加显式 `revision bigint` 或 `updated_at + editor_id` 字段，并用触发器或统一更新 SQL 维护。
- 删除要素接口也携带版本，避免用户删除已被别人修改过的要素。
- 在冲突面板中增加本地草稿与服务端版本的属性差异、几何差异预览。
- 增加“另存为新要素”“复制本地草稿”“确认覆盖保存”等冲突处理动作。
- 将冲突事件写入审计日志，便于排查多人编辑冲突高发图层。
