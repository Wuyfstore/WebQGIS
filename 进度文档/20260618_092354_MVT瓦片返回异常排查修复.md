# MVT瓦片返回异常排查修复

## 本次目标

排查 WebQGIS 的 MVT 矢量瓦片接口是否正常返回，并修复真实请求中出现的 500 错误。

## 做了哪些

- 直接请求 `/api/layers`，确认图层注册接口正常，6 个图层可读取。
- 对两个 SRID 3857 图层和一个 SRID 4326 图层直接请求 MVT 瓦片，发现均返回 500。
- 用相同数据源和相同 tile 坐标直接运行 PostGIS MVT SQL，确认 `ST_AsMVT` 能正常生成二进制瓦片。
- 启动临时 4101 API 实例抓取 Nest/Fastify 日志，定位根因：
  - `TypeError: response.setHeader is not a function`
  - Fastify reply 使用 `header()`，不是 Node response 的 `setHeader()`。
- 修改 `LayersController` 中 live/offline MVT 接口的 ETag 写入方式，兼容 Fastify `header()` 与 Node `setHeader()`。
- 新增 `layers.controller.spec.ts`，覆盖 live/offline MVT 接口在 Fastify reply 上写 ETag 并返回 Buffer。
- 重新请求真实 MVT URL，确认返回 200、content-type 为 `application/vnd.mapbox-vector-tile`，且二进制内容非空。

## 没做哪些

- 没有调整 MVT SQL、瓦片缓存、tileVersion 或离线瓦片发布逻辑。
- 没有改变前端 OpenLayers 加载逻辑。
- 没有重构控制器为流式响应，本轮只修复 Fastify 响应头兼容问题。

## 哪里有 bug

- 已修复：MVT 控制器使用 `response.setHeader()`，在当前 Fastify 适配器下该方法不存在，导致所有 MVT 请求 500。
- 本轮未发现 PostGIS MVT SQL 生成异常；同 tile 坐标直接执行 SQL 可得到非空 MVT。

## 建议后续做什么

- 为 MVT 接口增加 E2E 测试，覆盖 Fastify 真实 HTTP 响应、content-type、etag 和非空 Buffer。
- 若后续需要更大瓦片吞吐，可评估 `StreamableFile` 或 Fastify reply 直接发送 Buffer 的性能差异。
- 对瓦片大小过大的图层继续调优字段裁剪、zoom limit 和概化表策略。
